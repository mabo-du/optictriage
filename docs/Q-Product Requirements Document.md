# Product Requirements Document (PRD)
## **Project Name:** OpticTriage
**Document Version:** 1.0  
**Target Release:** MVP v1.0  
**Status:** Approved for Development  

---

## 1. Executive Summary
**OpticTriage** is a lightweight, non-destructive diagnostic utility designed to prepare, validate, and structure massive batches of photogrammetric imagery *before* they are ingested into heavy-duty Structure-from-Motion (SfM) pipelines (Agisoft Metashape, OpenDroneMap, COLMAP). 

Managing thousands of field photographs is chaotic. Uncurated datasets lead to exponential computational overhead, silent algorithmic failures, and catastrophic geometric distortions. OpticTriage acts as a critical filtration and normalization layer, saving researchers hours of manual sorting and preventing costly reconstruction crashes.

### The Core Philosophy: "Warnings, Not Deletions"
In field sciences (archaeology, palaeontology, heritage documentation), every photograph may contain irreplaceable contextual data. OpticTriage operates on a strict **non-destructive diagnostic framework**. It does not silently delete blurry or overexposed images; instead, it provides a quantitative "Health Dashboard," empowering scientists to make informed, expert-level decisions about their data.

## 3. User Experience (UX) & Workflow
OpticTriage translates complex computer vision mathematics into an accessible, visual workflow.

1. **Ingest & Triage:** The user drops a folder of 5,000 mixed RAW/JPEG images into the app. OpticTriage instantly extracts embedded previews and reads metadata in the background.
2. **The Health Dashboard:** The user is presented with interactive histograms. Instead of trusting a hidden, hardcoded algorithm, the user drags a slider across the "Blur Distribution" graph to visually set the threshold for what constitutes a "blurry" image based on their specific lens and lighting conditions.
3. **Target & Metadata Audit:** The app flags images containing ChArUco boards or ColorCheckers. It alerts the user if DJI drone altitude data is conflicting, offering a one-click safe fix.
4. **Platform Handoff:** The user selects their downstream software (Metashape, ODM, or COLMAP). OpticTriage generates a perfectly structured, ready-to-run project folder.

## 4. Technical Architecture & Pipeline
The application is modeled as a **Sequential Processing Graph**. Each stage is independent, allowing users to re-run specific steps (e.g., re-evaluating blur thresholds) without re-processing the entire dataset from scratch.

### 4.1. The RAW Performance Bypass
Decoding full-resolution RAW sensor data (demosaicing) for thousands of images is computationally prohibitive for a lightweight tool. 
* **The Solution:** OpticTriage bypasses the RAW array entirely by extracting the **embedded JPEG preview** generated by the camera's internal Image Signal Processor (ISP). 
* **Implementation:** Utilizing `rawpy` (LibRaw) and `ExifTool` subprocess streaming, the tool extracts high-resolution previews in milliseconds. These previews are geometrically identical to the RAW file and perfectly preserve the high-frequency spatial data required for blur detection and fiducial marker identification.

### 4.2. State Management
To ensure crash resilience on massive datasets, OpticTriage utilizes a localized **SQLite database**. 
* **`Session` Table:** Tracks input/output paths, global settings, and processing state.
* **`ImageRecord` Table:** Tracks the exact state, quality scores, EXIF data, and detected targets for every individual image. If the app is interrupted at image #4,500 out of 5,000, it resumes exactly where it left off.

## 5. Feature Specifications (MVP v1.0)

### 5.1. Multidimensional Quality Scoring
OpticTriage evaluates image health using a composite scoring framework, avoiding the false positives of global metrics.
* **Patch-Based Blur Detection:** Instead of global Laplacian Variance (which fails on sharp subjects with blurry backgrounds), the image is subdivided into a grid. The tool evaluates the top 5% sharpest patches to determine if the *subject* is in focus.
* **Veiling Glare Detection:** Converts images to HSI (Hue, Saturation, Intensity) color space to identify localized bright spots with low saturation (sun flare/lens scatter) that deprive SfM algorithms of distinct feature points.
* **Exposure Clipping:** Analyzes luminance histograms to flag images with crushed shadows or blown-out highlights (>1% of pixels clipped at 0 or 255).

### 5.2. Fiducial & Target Detection (Precision over Recall)
Field conditions are hostile. The system must prioritize **high precision** (certainty) over recall to prevent a rock or shadow from being misidentified as a control point, which would mathematically ruin the 3D map.
* **Preprocessing Pipeline:** Images are converted to LAB color space. **CLAHE** (Contrast Limited Adaptive Histogram Equalization) is applied to the Lightness channel to reveal markers hidden in deep trench shadows, followed by Bilateral Filtering to preserve sharp marker edges.
* **ArUco & ChArUco:** Utilizes OpenCV's `cv2.aruco` module. ChArUco is prioritized for its resilience to severe occlusion (mud/partial framing).
* **Sub-Pixel Refinement:** Executes `cv2.cornerSubPix` to achieve the sub-millimeter accuracy required for professional photogrammetric scaling.
* **ColorChecker (X-Rite):** Uses `cv2.mcc` to detect 24-patch charts. **Crucial Rule:** The tool *flags* these images for the user but does *not* auto-apply a Color Correction Matrix (CCM). Applying a single CCM across a dataset with shifting cloud cover causes severe spectral distortion.

### 5.3. Metadata Integrity & The Drone Altitude Fix
Missing or conflicting EXIF/XMP data is a primary cause of silent pipeline failures.
* **The DJI Altitude Bug:** Consumer drones record highly inaccurate `GPSAltitude` (ellipsoidal) in standard EXIF, but hide highly accurate barometer-derived `RelativeAltitude` in proprietary XMP namespaces (`XMP:drone-dji`).
* **The Fix:** OpticTriage detects DJI files, extracts the precise XMP Relative Altitude, and uses an **ExifTool subprocess** to safely overwrite the flawed EXIF `GPSAltitude` tag. Subprocess execution is mandatory to prevent pure-Python libraries (like `piexif`) from corrupting fragile camera MakerNotes.
* **Mandatory Tag Validation:** Asserts the presence of `FocalLength`, `SensorWidth`, and valid GPS bounds. Halts and alerts the user if critical intrinsics are missing.

### 5.4. Platform-Specific Exporters
OpticTriage abandons a "universal" manifest in favor of bespoke, syntactically perfect exporters for the big three SfM engines.

| Target Software | Export Strategy | Key Technical Requirements |
| :--- | :--- | :--- |
| **Agisoft Metashape** | **Headless Python Script Generator** | Metashape's `.psx` binary format is proprietary and prone to corruption. OpticTriage exports a `.py` script utilizing the `Metashape.app.document` API to programmatically instantiate the project, ingest photos, set the CRS, and import GCP CSVs. |
| **OpenDroneMap (ODM)** | **Structured Directory & Text Manifests** | Generates `gcp_list.txt` (whitespace delimited, PROJ/EPSG header, **strictly replacing `NaN` Z-values with `0.0`** to prevent matrix crashes) and `cameras.json` (OpenSfM Brown-Conrady models). |
| **COLMAP** | **Pre-populated SQLite & INI** | Initializes a `database.db` via Python's `sqlite3`. Enforces `camera_id > 0` and `image_id > 0`. Generates `cameras.txt`, `images.txt`, and a `project.ini` configuration file for immediate CLI execution. |

## 6. Proactive Quality Assurance (Failure Mode Traps)
OpticTriage acts as a vigilant scout, deploying algorithmic traps to catch edge cases before they reach the SfM software.

| Failure Mode | Trigger Mechanism | OpticTriage Algorithmic Trap |
| :--- | :--- | :--- |
| **Zero-Baseline Duplicates** | Shutter burst or copy errors. | Compute **MD5/SHA-256 cryptographic hashes** of the image data stream. Automatically quarantine exact duplicates to prevent matrix inversion crashes in bundle adjustment. |
| **Resolution Mismatches** | Mixed 4K video frames and 20MP stills. | Parse `ImageWidth`/`ImageLength`. Assert uniform dimensions per processing block. Halt and alert if deviations exist. |
| **Memory Overload** | 10,000x 45MP images on 16GB RAM. | Estimate RAM footprint based on image count/resolution. Warn user and suggest logical chunking before they attempt alignment. |
| **Null Elevation Vectors** | RTK rovers losing fix, exporting `NaN`. | Regex parser over GCP data. Auto-replace spatial `NaN` with `0.0` to save ODM from total georeferencing failure. |

## 7. Technology Stack
* **Core Language:** Python 3.10+
* **GUI Framework:** PyQt6 (Desktop-native, workflow-style UI)
* **Computer Vision:** OpenCV (ArUco, CLAHE, Laplacian, MCC)
* **Metadata Engine:** ExifTool (via `subprocess` for safe writes), `piexif` (for fast reads)
* **RAW Handling:** `rawpy` (LibRaw wrapper)
* **Database:** SQLite (via `sqlite3`)
* **Packaging:** PyInstaller / Tauri (for lightweight cross-platform distribution)

## 8. Future Roadmap (v2.0+)
* **GPS Track Alignment:** Import `.gpx` drone flight logs and align with image timestamps via linear interpolation for superior georeferencing.
* **Perceptual Hashing (pHash):** Move beyond exact MD5 duplicates to flag "near-duplicates" (insufficient baseline for stereo reconstruction).
* **Camera Group Auto-Segmentation:** Automatically split massive datasets into logical processing chunks based on time gaps, altitude shifts, or GPS clustering.
* **Dynamic Color Normalization:** For controlled studio environments, apply keyframe-based CCMs to localized image subsets.
